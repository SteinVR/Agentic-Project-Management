# Идея проекта

Стандартизация разработки проектов по методологии.
- Систематизация разработки проектов на основе нескольких методологий.
- Упрощение работы с SDD в среде Cursor.

# Тело проекта

1. CLI - Конфигуратор, создающий шаблон (директорию проекта в указанной директории с указанным именем) для начала работы над проектов в среде Cursor.
	1. Создание директории
	2. Создание рабочей среды
		- .apm c соответствующими папками в зависимости от выбранной методологии
		- commands для Cursor
		- дополнительные скрипты
	3. Создание соответствующего репозитория git и синхронизация с github.
	4. Открытие Cursor в папке проекта.
2. Автоматизация внутри Cursor за счет команд и подготовленных шаблонов.

# Interactive Flow
## Взаимодействие с конфигуратором

1. Пользователь открывает cli скрипт. 
2. Запускается терминал с оформленным cli ui и приветствием
3. Пользователю предлагается выбрать директорию, где будет лежать проект
4. Пользователю предлагается ввести имя проекта
5. Пользователю предлагается авторизация в github (если она не была проведена - нужно проверку интегрировать) для последующего создания удаленного репозитория. Он может отказаться.
6. Пользователю дается краткое описание каждой из методологий и предлагается выбрать методологию
7. После подтверждения корректности директории и имени, запускается процесс, в котором:
	1. Создается именная папка в указанной директории
	2. В папке создается директория (.apm/)для агентов и шаблонов с учетом выбранной методологии
	3. Создается папка для cursor (.cursor/), туда копируются скрипты.
	4. Создается локальный git
	5. Если пользователь согласен с созданием удаленного репозитория и авторизация была успешно завершена -> создание удаленного репозитория и первая синхронизация с локальным репозиторием. Если нет -> фаза пропускается.
8. Появляется уведомление о окончании фазы подготовки и "проект готов к работе". Предлагается открыть Cursor в папке проекта автоматически. Пользователь может отказаться.

## Процесс работы над проектом внутри Cursor:
1. Пользователь пишет команду (/apm-start) и высокоуровнево описывает идею проекта. В идеале, если он опишет Идею проекта, Тело проекта и Interaction Flow (step by step).
2. Архитектор после получения информации о проекте от пользователя, его "видения", перед построением файла архитектуры, должен его проанализировать и описать так, как он его понял (для валидации) в формате:
	- Идея проекта (ядро, философия, применимость, потребность, видение)
	- Тело проекта - в каком виде проект будет реализован:
		- Приложение (cli, web, ios, android)
		- Конфигуратор (.sh, .bat)
		- т.д.
	- Последовательное описание принципа работы проекта/сценарий работы (как видит процесс работы пользователь)(step-by-step narrative of how the user interacts with the finished product).
	Если от пользователя чего-то не хватило для полноценного заполнения (условно например тела проекта или step-by-step narrative), то агент заполняет эти поля самостоятельно от своего видения проекта, а после просит пользователя ознакомиться с ними и поделиться с ним корректировками, если видение отличается.
	Кроме того, вносит предложение по усовершенствованию идеи или новые идеи, моменты, которые пользователь мог упустить. Т.е. попробовать дополнительно развить идею или предложить новую на основе старой. (В идеале добиться эффекта: "Вау, вот это идея!")
3. Архитектор на основе всей информации составляет файл архитектуры, и прочие файлы в зависимости от выбранной методологии.
4. Идет работа над проектом с Developer. В зависимости от загруженности контекстного окна в чате, общение может вестись как в одном окне весь проект, так может и создаваться новые окна и продолжение работы через инициализацию агента Developer команой (/apm-develop)
5. Кроме того, могут вызываться специфические отчеты командой (/apm-report) или "осмотр" текущего состояния проекта командой (/apm-review)

# Комманды

- /apm-start - Инициализация проекта. Агенту cursor присоединяется файл роли архитектора, команда ей следовать и доп. инструкция. После нее ожидается описание проекта пользователем.
- /apm-develop - Вызов агента разработчика. Агенту cursor прикрепляется файл роли разработчика, указание ей следовать, ознакомиться с проектом. После нее можно внести специфические указанию разработчику или ничего не указывать и просто запустить команду в окне чата, в таком случае агент должен подхватить актуальные задачи.
- /apm-report - Команда для запроса отчета. В чат прикрепляется файл с инструкцией по написанию отчета. 
- /apm-review - Команда для обзора текущего проекта как высокоуровнево, так и на уровне отдельных компонентов. Агенту cursor прикрепляется файл роли архитектора.
  Цель - осмотреть проект свежим взглядом, его структуру, принцип работы. Выявить текущие, потенциальные, критические проблемы, отклонения от изначальной идеи, архитектуры, появившийся тех. долг (если есть). Написание соответствующего отчета. 

# Методологии
## Full Methodology

Full использует исчерпывающую методологию, с полной системой документации, и разделения компонентов проекта по отдельным блокам, в которых изолированно (с возможностью распараллеливания) происходит реализация функционала через отдельные файлы task.md. 

Данная методология будет лучше всего работать в больших проектах, приложениях, где есть или планируется front, где много back end логики, микросервисы и т.д. Где кодовая база достаточно большая, чтобы оправдать используемые ограничения для агентов в методологии, и количество вызовов агентов. 

Методология пародирует полноценные отдельные команды разработчиков и здоровые процессы в ней.

## Rapid Methodology
Rapid методология предназначена для более быстрой разработки проектов, находящихся на этапе MVP или стремящихся к ней.

Здесь важна более быстрая и удобная развертка нового функционала, чем исчерпывающий контроль тестов и строгий регламент. Здесь не планируется использовать отдельные папки под каждый компонент проекта (единая папка для исходников проекта), но разработка все так же использует философию блочности и абстракции (использование классов, интерфейсов, методов, отдельных функций вне, строгой типизации) для гибкости, модульности и опять же возможности параллельной разработки несколькими разработчиками без необходимости им взаимодействовать друг с другом.

В данной методологии нет необходимости пародировать отдельные комманды разработчиков. Разработка происходит уже больше в ручном режиме с юзером, который отдает команды сначала архитектору, а после стадии планирования уже с агентом разработчика и тестером.
